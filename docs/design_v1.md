## 1. 專題目標

### 1.1 一句話定義

給定一段「自然語言描述的事由」（例如：「我開車不小心擦撞到路邊停靠的貨車，對方後照鏡損壞」），系統從台灣刑事判決書資料庫中，找出**事實相似**的歷史案例，並彙整呈現「過去法院怎麼判、賠償範圍、為什麼」。

### 1.2 使用者輪廓

法律門外漢。不懂法條、不會用法律術語搜尋，只知道「發生了什麼事」。系統要能從口語事由出發，跨越「口語 ↔ 法律術語」的鴻溝。

### 1.3 系統定位（重要）

本系統是**類案檢索與資訊呈現工具**，不是法律建議系統。

- 做：呈現「過去類似案例的事實與判決結果」，用語為「過去法院曾經…」
- 不做：對使用者的特定個案做預測或建議（「你會被判…」）

理由見第 6 節（法律正當性）。

## 2. 資料分析（對照真實資料的可行性驗證）

資料位置：`ssh home_wsl:/home/mrfrog/code/lawundry_test/Dataset`

### 2.1 資料結構

- 158 個頂層批次資料夾（依年月更新打包），下分院別／審級資料夾
- 每份判決是一個 JSON，欄位僅 8 個：
  - `JID`：判決書唯一識別碼
  - `JYEAR` / `JCASE` / `JNO`：年度 / 案件類別 / 案號
  - `JDATE`：裁判日期
  - `JTITLE`：案由（如「過失傷害」「公共危險」「毀損」）
  - `JFULL`：判決全文（核心，數百至上萬字）
  - `JPDF`：多為空字串
- 真正有內容的只有 `JTITLE` 與 `JFULL`。**刑度、賠償金額、責任比例都埋在 JFULL 自然語言裡，需靠抽取。**

### 2.2 實測統計（抽樣驗證）

| 項目 | 數值 | 來源 |
|------|------|------|
| 刑事判決 JSON 總數 | **532,276 筆** | 全量掃描 |
| 符合目標案由（過失/毀損/公共危險）比例 | **22.2%** | 抽樣 3000 份 |
| 推估目標子集量級 | **10 萬筆以上** | 推估 |
| 目標案件結構三段齊全（主文+事實+理由） | **約 87%** | 抽樣 |

Top 案由（抽樣）：公共危險 > 竊盜 > 毒品 > 詐欺 > 過失傷害…

### 2.3 關鍵限制（必須在報告與 UI 誠實交代）

**限制 1 — 資料量過大**：53 萬刑事筆數無法全量 embed。需縮減子集（見 3.1）。

**限制 2 — 長文本**：依 Peng & Lei (PeerJ CS 2024)，85.43% 台灣判決書超過 BERT 512 token 上限，本系統實測吻合。需段落切割。

**限制 3 — 無 ground truth**：無人工標注的 (query, relevant_doc) 對。需自建合成評估集（見第 5 節）。

**限制 4 — 選擇性偏差（Selection Bias，最重要）**：
能進入「刑事判決書」的車禍案件，幾乎都涉及「人員受傷（過失傷害）」或「肇事逃逸（刑法 185-4）」。
**純財物損害（如僅後照鏡損壞）在刑事上通常不起訴**——因刑法第 354 條毀損罪以「故意」為要件，過失毀損不罰，這類糾紛多走民事或調解，不會留下刑事判決。
實測：目標案件中「不受理／公訴不受理」72 件、「無罪」7 件，且搜尋「過失+毀損+不罰」跳出的多為妨害名譽、公平交易法案件，而非車禍財損。

**限制 5 — 資料庫只有刑事**：使用者情境常是民刑交界，系統僅能涵蓋刑事面，民事賠償責任面需明確告知不在範圍。

## 3. 系統架構

### 3.1 離線預處理階段（一次性）

```
53 萬刑事判決
   │
   ├─[Step 1] 篩選子集
   │   只取：過失傷害 + 公共危險(交通類) + 肇事逃逸，限地方法院
   │   目標：壓到 1–3 萬筆
   │
   ├─[Step 2] 結構切段（regex）
   │   切「主文 / 事實(或犯罪事實) / 理由」三段
   │   重點處理「事實段」；13% 結構不全者走 fallback
   │
   ├─[Step 3] 四層要素抽取（見 4 節）
   │   輸出結構化 JSON：facts_summary / cited_articles /
   │   verdict / sentence / compensation / key_factors / confidence
   │
   ├─[Step 4] 事實段加結構化標頭後 embed
   │   標頭範例：[法院:士林地院 | 案由:過失傷害 | 年度:108 | 字號:交訴5]
   │   Embedding：BGE-M3 → ChromaDB
   │
   └─[Step 5] 同時建 BM25 index（法條號碼精確比對用）
```

### 3.2 線上查詢階段

```
口語事由輸入
   │
   ├─[A] Query Understanding（LLM）
   │   提取：事故類型(財損/傷亡)、主觀要素(過失/故意)、可能法條列表
   │   口語→術語正規化（撞到→碰撞致損、賠錢→損害賠償）
   │   （註：不做 HyDE 假設文件改寫）
   │
   ├─[B] 雙路檢索
   │   B-1 Dense：BGE-M3 查事實段 → top-20
   │   B-2 Sparse：BM25 用法條號碼查全文 → top-20
   │
   ├─[C] RRF 融合（k=60）→ top-10
   │
   ├─[D] Reranking：bge-reranker-v2-m3 → top-5
   │
   ├─[E] Answer Synthesis（LLM，帶 citation）
   │   每個結論標注「依據 JID 某案第幾段」
   │
   ├─[F] Verification：檢查 LLM 引用的法條是否真在原文
   │   未出現者標紅（citation grounding 輕量版）
   │
   └─[G] 加上資料偏差免責說明後輸出
```

## 4. 要素抽取：四層可驗證管線（比單用 LLM 更公正）

單一次 LLM 抽取不可驗證、易幻覺。採組合拳：

| Layer | 做法 | 工具 | 目的 |
|-------|------|------|------|
| L1 結構約束 | Pydantic schema 強制合法 JSON | Outlines / Instructor | 杜絕結構幻覺 |
| L2 規則交叉驗證 | 法條號/刑度/日期/案號用 regex 抽 | LexNLP / regex | 固定格式不交給 LLM |
| L3 一致性投票 | 判斷型欄位（過失/故意）LLM 跑 3–5 次多數決 | self-consistency | 產生信心分數 |
| L4 忠實度檢查（選做） | NLI 檢查欄位值是否被原文蘊涵 | DeBERTa-NLI | 標記潛在幻覺 |

**交叉驗證規則**：LLM 抽出法條 ∩ regex 抽出法條 = 高信心；單邊有 = 標記待人工複核。

**學術價值（消融實驗）**：比較「純 LLM 抽取」vs「LLM+規則交叉驗證」的準確率與幻覺率，呼應課程「可信度」主題。

## 5. 評估設計

### 5.1 合成評估集

從篩選後子集隨機抽 ~50 份判決，對每份用 LLM 生成 2–3 個「法律門外漢可能會問的口語事由」，得到約 100–150 個 (query, relevant_doc) 對。

### 5.2 指標

- **Recall@5**：前 5 結果是否含 relevant doc，目標 > 0.6
- **MRR**：首個命中平均倒數排名，目標 > 0.4
- **消融實驗**：Dense vs BM25 vs Hybrid vs Hybrid+Rerank
- **RAGAS faithfulness**：自動偵測法條幻覺率
- **要素抽取對比**：純 LLM vs LLM+規則 的準確率 / 幻覺率（對齊 CAIL2019-FE 的 Micro/Macro-F1）

## 6. 可信度與法律正當性（不踩雷）

### 6.1 法律地雷

1. **禁止個案預測/建議**：可能觸及《律師法》非律師執行法律業務。只呈現過去案例，用詞「過去法院曾經…」。
2. **資料偏差須主動揭露**：純財損找不到刑事案例，系統須主動說明（見限制 4）。
3. **去識別化**：判決原文已有 ◯◯ 遮蔽，展示時不還原、不交叉比對當事人。
4. **資料授權**：來源為司法院開放資料（opendata.judicial.gov.tw，本即開放）。**非自行爬取法源/Lawsnote**（Lawsnote 創辦人因爬蟲一審遭判 4 年、賠 1 億，為真實判例，務必避免）。

### 6.2 技術可信度

核心原則 **Citation Grounding**：每個結論可指回原文。

- 法條由 BM25 在原文精確比對（確保存在），不讓 LLM 自由生成
- 每個摘要結論標注「依據 JID 某案第幾段」
- Verification 步驟檢查引用法條是否真在判決全文，未出現標紅
- 依據：Citation Grounding 研究（2025）指出商用 LLM 法律引用有 13–21% 為幻覺

## 7. 技術選型

| 元件 | 選擇 | 理由 |
|------|------|------|
| Embedding | BGE-M3 | 單模型同出 dense+sparse，中文最佳，本地免費 |
| Reranker | bge-reranker-v2-m3 | MIT 授權、多語言、8192 上下文 |
| Vector DB | ChromaDB | 輕量、本地 |
| Sparse 檢索 | BM25（BM25s） | 法條號碼精確比對 |
| 融合 | RRF（k=60） | 只用排名免 normalization |
| 結構化抽取 | Outlines / Instructor + Pydantic | 約束解碼保證合法 JSON |
| 規則抽取 | LexNLP / regex | 法條號/刑度/金額 |
| LLM | GPT-4o-mini / Claude Haiku 等 | 抽取與摘要，便宜夠用 |
| 評估 | RAGAS | reference-free，可測 faithfulness |
| Python 套件管理 | uv | 專案規範 |

## 8. 設計取捨

| 選項 | 決策 | 理由 |
|------|------|------|
| Graph RAG | 不做 | 工程量大、關係難定義，期末來不及；改用「法條節點」當 BM25 橋樑 |
| HyDE 假設文件改寫 | 不做 | 依使用者決定，改用術語正規化處理口語落差 |
| Fine-tune 模型 | 不做 | LLM prompt + 規則已足夠，時間不足 |
| 全文 embed | 不做 | 失焦，改 embed 事實段 |
| 法條 embed | 不做 | 法條用 BM25 精確比對，不需語意模糊 |
| 全量 53 萬筆 | 不做 | 篩 1–3 萬筆交通相關刑事 |
| 民事判決 | 不做 | 資料集僅刑事，不混 |
| 實時 LLM 抽取 | 不做 | 改離線預處理，降低 query 延遲 |

## 9. 可引用文獻地圖

| 設計決策 | 引用文獻 | 證明了什麼 |
|---------|---------|-----------|
| 事由→法條→案例 階層 | TopJudge, Zhong et al., EMNLP 2018（thunlp/TopJudge） | 階層式法律預測有效；罪名 Micro-F1 97.6% |
| 檢索應加入法條 | NyayaRAG, IJCNLP-AACL 2025（ShubhamKumarNigam/NyayaRAG） | 加法條 +15%，再加判例反降（雜訊） |
| 判決書需切段 | Peng & Lei, PeerJ CS 2024, DOI:10.7717/peerj-cs.1841 | 85.43% 台灣判決超 512 token |
| 中文長文 embedding | Lawformer, arXiv:2105.03887（thunlp/LegalPLMs） | 中文法律長文預訓練模型（4096 token） |
| 要素抽取任務化 | CAIL2019-FE | 要素抽取標準評估（Micro/Macro-F1） |
| LLM+規則混合抽取 | SemEval-2023 LegalEval, ACL 2023.semeval-1.173 | 規則集成 F1 90.39% 略勝純 LLM 90.27% |
| citation 驗證防幻覺 | Citation Grounding, arXiv 2025 | LLM 法律引用 13–21% 為幻覺 |
| 結構化 JSON 抽取 | JSONSchemaBench, arXiv:2501.10868 | 約束解碼保證合法且不傷品質、最高 +4% |
| 注入上下文提升檢索 | Anthropic Contextual Retrieval, 2024 | 檢索失敗率降 49–67% |
| 評估框架 | RAGAS | reference-free，faithfulness 偵測幻覺 |

**優先精讀（按重要性）**：
1. NyayaRAG — 直接驗證核心架構（法條要不要加）
2. Peng & Lei 2024 — 台灣同源資料、長文處理
3. SemEval-2023 LegalEval — 支持混合抽取與消融實驗設計
